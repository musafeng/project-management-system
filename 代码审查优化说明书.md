## 一、🔴 严重问题（建议立即修复）

### 1.1 数据库密码泄露在 `.env` 中且可能已入 Git 历史

| 项目 | 详情 |
|------|------|
| 文件 | `.env`（根目录） |
| 问题 | 文件中包含完整的 Supabase 数据库连接字符串（含明文密码）。虽然 `.gitignore` 已加入 `.env`，但文件**目前仍存在于工作目录中**，且可能已经进入了 Git 历史。 |
| 建议 | ① 立即轮换（rotate）该数据库密码 ② 用 `git filter-branch` 或 BFG Repo-Cleaner 清除历史 ③ 确认 `.env` 确实未被 commit |

### 1.2 认证 Cookie 未签名/未加密 — 用户可伪造身份

| 项目 | 详情 |
|------|------|
| 文件 | `lib/auth.ts:38`、`lib/api/auth.ts:37` |
| 问题 | 用户信息（userid、name 等）被**明文 JSON 序列化后直接写入 Cookie**，仅使用 `httpOnly + secure` 保护。恶意用户可以通过浏览器控制台手动构造 `auth` Cookie，伪造为任意用户（包括 ADMIN）。 |
| 建议 | 使用 JWT（加签名密钥）或 `iron-session` 等库对 Cookie 内容进行**签名 + 加密**。 |

### 1.3 SQL 注入风险 — `db-column-compat.ts` 中表名直接拼接

| 项目 | 详情 |
|------|------|
| 文件 | `lib/db-column-compat.ts:22` |
| 问题 | `$queryRawUnsafe` 中直接将 `tableName` 拼入 SQL：`...table_name = '${tableName}'`。虽然调用来自内部，但 `db-write-compat.ts` 中已有 `assertIdentifier` 校验函数，此处却未使用。 |
| 建议 | 添加 `assertIdentifier(tableName)` 校验，或改用 `Prisma.sql` 参数化查询。 |

### 1.4 上传 API 缺少认证 — 任何人可上传文件到 OSS

| 项目 | 详情 |
|------|------|
| 文件 | `app/api/upload/route.ts` |
| 问题 | 上传接口没有任何认证校验（未使用 `apiHandlerWithPermission` 或 `getCurrentUser`）。 |
| 建议 | 至少添加 `requireAuth()` 认证检查。 |

## 二、🟠 重要问题（建议近期优化）

### 2.1 API 错误处理逻辑高度重复 — 3 个 handler 文件 90% 代码相同

| 项目 | 详情 |
|------|------|
| 文件 | `lib/api/handler.ts`、`handler-with-permission.ts`、`handler-with-log.ts` |
| 问题 | 三个文件的 `catch` 块逻辑几乎完全相同（ApiError → 禁用 → 未登录 → 无权限 → Unique → Foreign key → 兜底），约 60 行 × 3 = **180 行冗余**。 |
| 建议 | 抽取 `handleApiError(err): NextResponse` 公共函数到 `errors.ts`，三个 handler 共用。 |

### 2.2 `permission-wrapper.ts` 与 `handler-with-permission.ts` 功能完全重叠

| 项目 | 详情 |
|------|------|
| 文件 | `lib/api/permission-wrapper.ts`、`lib/api/handler-with-permission.ts` |
| 问题 | 两者都提供权限包装功能（`withApiPermission` vs `apiHandlerWithPermission`），接口定义重复（`PermissionCheckOptions` vs `PermissionOptions`）。 |
| 建议 | 统一为一套，删除不再使用的那个。 |

### 2.3 `AuthUser` 类型定义重复 — 3 处不同步

| 项目 | 详情 |
|------|------|
| 文件 | `lib/auth.ts:11`、`lib/auth-client.ts:9`、`lib/api/auth.ts:14` |
| 问题 | 三处各自定义了用户接口，字段略有差异（`auth-client.ts` 多了 `systemRole`、`isActive`）。 |
| 建议 | 统一到一个共享的 `lib/types/auth.ts` 文件。 |

### 2.4 钉钉 `access_token` 未缓存 — 每次请求都重新获取

| 项目 | 详情 |
|------|------|
| 文件 | `lib/dingtalk.ts` `getAccessToken()` |
| 问题 | 钉钉 access_token 有效期 2 小时，但每次调用钉钉 API 都重新请求 token，高频调用会触发限流。 |
| 建议 | 加内存缓存（Map + TTL 1.5 小时），过期后自动刷新。 |

### 2.5 Prisma query log 在生产环境仍然开启

| 项目 | 详情 |
|------|------|
| 文件 | `lib/prisma.ts:8` |
| 问题 | `log: ['query']` 始终开启，**生产环境也会打印所有 SQL**，影响性能且可能泄露数据。 |
| 建议 | 改为 `log: process.env.NODE_ENV === 'development' ? ['query'] : []` |

### 2.6 缺少 API 输入校验库

| 项目 | 详情 |
|------|------|
| 文件 | 所有 API route 的 POST/PUT |
| 问题 | 靠 `if (!body.name ...)` 手动逐个字段校验，金额/日期字段无类型和范围校验。 |
| 建议 | 引入 `zod` 进行 schema 校验，前后端复用。 |

### 2.7 XSS 风险 — `dangerouslySetInnerHTML`

| 项目 | 详情 |
|------|------|
| 文件 | `app/approval/page.tsx` |
| 问题 | 使用了 `dangerouslySetInnerHTML`，需确认内容是否经过消毒。 |
| 建议 | 使用 `DOMPurify` 对 HTML 内容消毒后再渲染。 |

## 三、🟡 前端代码质量（33 个页面、8 个组件）

### 3.1 20+ 页面重复相同的"外壳"结构 — 已有抽象未推广

| 项目 | 详情 |
|------|------|
| 文件 | `app/customers/page.tsx`、`app/procurement-contracts/page.tsx`、`app/labor-contracts/page.tsx` 等 20+ 页面 |
| 问题 | 每个页面都重复着完全相同的骨架代码（ConfigProvider + 容器 div + 标题 + 搜索栏 + Table + Modal），约 **400 行纯重复代码**。**但项目中已有 `components/ledger/LedgerPageLayout` + `FilterBar` + `StatusTag` 这套优秀抽象，却只有少数新页面在使用。** |
| 建议 | 将全部旧页面迁移到 `LedgerPageLayout` + `FilterBar`，可一次性消除 **60%+ 重复代码**。 |

### 3.2 `ConfigProvider` 在每个页面各自挂载 — 主题配置重复 20+ 次

| 项目 | 详情 |
|------|------|
| 文件 | 几乎每个旧页面 |
| 代码 | `<ConfigProvider theme={{ token: { colorPrimary: '#1677ff', borderRadius: 6, fontSize: 14 } }}>` |
| 建议 | 移至根布局 `app/layout.tsx`，全局生效一次即可。改主色只需改一处。 |

### 3.3 工具函数重复定义 — `formatDate`/`formatCurrency`/`ApiResponse` 各页面各写一遍

| 项目 | 详情 |
|------|------|
| 问题 | `formatDate()` 在 10+ 个文件各自定义一遍；`formatCurrency()` 同样；`interface ApiResponse<T>` 在 10+ 个文件逐字重复。**`lib/utils/format.ts` 中已有 `fmtMoney` 和 `fmtDate`，但绝大多数页面没有导入。** |
| 建议 | 统一使用 `lib/utils/format.ts`，删除各页面局部定义。`ApiResponse<T>` 移入 `lib/types.ts`。 |

### 3.4 状态映射表不一致 — `STATUS_MAP` 定义了 5+ 份，颜色值还不同

| 项目 | 详情 |
|------|------|

| 建议 | 全部统一使用 `StatusTag` 组件。 |

### 3.5 `ApprovalActions` 组件每表格行各发一次请求 — N+1 问题

| 项目 | 详情 |
|------|------|
| 文件 | `components/ApprovalActions.tsx:59-76` |
| 问题 | 每个 `approvalStatus === 'PENDING'` 的表格行都在 `useEffect` 中独立发起 `/api/process-tasks/pending?...` 请求。50 条记录 = 50 个并行请求。 |
| 建议 | 提供批量查询接口，一次性获取当前用户可审批的所有 ID 集合；或随列表数据一起返回审批权限。 |

### 3.6 `app/projects/page.tsx` 使用 `window.location.href` 而非 `useRouter`

| 项目 | 详情 |
|------|------|

### 3.7 项目详情页一次性发 6 个 API — 无 Tab 懒加载

| 项目 | 详情 |
|------|------|
| 文件 | `app/projects/[id]/page.tsx:330-345` |

### 3.8 `useEffect` 依赖项被人为跳过 — 过期闭包风险

| 项目 | 详情 |
|------|------|
| 文件 | `app/approval/page.tsx:264` |
| 问题 | `useEffect(() => { load() }, [tab]) // eslint-disable-line` — `load` 是 `useCallback`，依赖 `tab`、`resourceType`、`keyword`，但 effect 只监听 `tab`，`resourceType`/`keyword` 改变时不会重新触发。 |

| 项目 | 详情 |

| 建议 | `columns` 用 `useMemo`；事件函数用 `useCallback`；DynamicForm 的 `sorted` 也需要 `useMemo`。 |

### 3.11 内联样式 + 硬编码颜色 — 无法主题化

| 项目 | 详情 |
|------|------|
| 问题 | 全项目大量内联 `style={{...}}`，颜色硬编码 100+ 处（`#52c41a`、`#ff4d4f`、`#1677ff`...），Tailwind 已安装但几乎未用。暗色模式/主题切换不可能实现。 |
| 建议 | 短期用 `theme.useToken()` 替代硬编码颜色；长期统一迁移 Tailwind 或 CSS Modules。 |

### 3.12 关键词搜索前后端策略不一致

| 项目 | 详情 |

## 四、🟢 数据库 Schema 优化建议

### 4.1 审批字段在 14 个模型中重复 — `approvalStatus` 默认值不一致

| 项目 | 详情 |
|------|------|
| 问题 | `approvalStatus`、`submittedAt`、`approvedAt`、`rejectedAt`、`rejectedReason`、`attachmentUrl` 在 14 个模型中重复。部分默认 `"APPROVED"`，部分默认 `"PENDING"`，易混淆。 |
| 建议 | ① 统一默认值 ② 在代码层抽取类型和校验 |

### 4.2 `approvalStatus` 应使用 Enum 而非 String

| 项目 | 详情 |
|------|------|
| 问题 | 审批状态是有限集合（APPROVED/PENDING/REJECTED），但用了自由 String，无法保证数据一致性。 |
| 建议 | 创建 `enum ApprovalStatus { PENDING APPROVED REJECTED }` 统一使用。 |

### 4.3 缺少 `regionId` 索引

| 项目 | 详情 |
|------|------|
| 问题 | `ContractReceipt`、`OtherReceipt` 等模型有 `regionId` 字段但无索引，工作台按 `regionId` 高频过滤。 |
| 建议 | 添加 `@@index([regionId])`。 |

### 4.4 缺少复合索引 — 常见查询模式未覆盖

| 项目 | 详情 |
|------|------|
| 问题 | 工作台同时按 `regionId + status` 或 `regionId + 日期` 查询，但仅有单列索引。 |
| 建议 | 添加 `@@index([regionId, approvalStatus])`、`@@index([regionId, paymentDate])` 等。 |

### 4.5 级联删除策略不一致

| 项目 | 详情 |
|------|------|
| 问题 | `FormField → FormDefinition` 有 `onDelete: Cascade`，但 `ContractReceipt → ProjectContract` 没有。删除合同时会因外键约束失败。 |
| 建议 | 核心数据用 `Restrict` + 业务层检查，附属数据用 `Cascade`，并文档化。 |
